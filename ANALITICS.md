# TravLight · Google Analytics 4 계측 문서

랜딩 → 서비스 진입 → 정보 탐색으로 이어지는 사용자 행동을 측정하기 위한 GA4 연동 정리 문서.
코드 구현, 수집 이벤트, GA4 설정, 분석 방법, 1차 실험 결과까지 한곳에 정리한다.

---

## 1. 실험 가설

> 해외여행을 준비하는 2030세대는 비자·세관·날씨·현지 꿀팁 정보를 탐색하며 시간 소모와 신뢰 불안을 겪는다.
> 국가별 정보를 공식 출처와 함께 한 화면에 모아 보여주면,
> ① 랜딩을 본 사용자 중 일정 비율이 **"TravLight 이용하기"** 를 눌러 서비스로 진입하고,
> ② 진입한 사용자는 여행 계획 국가의 정보 **탭을 클릭**해 탐색할 것이다.

이를 **도달 → 전환 → 참여** 3단계 퍼널로 측정한다.

| 단계 | 의미                     | 지표(이벤트)            |
| ---- | ------------------------ | ----------------------- |
| 도달 | 랜딩 페이지 방문         | `page_view` / 방문자 수 |
| 전환 | "이용하기"로 서비스 진입 | `cta_enter`             |
| 참여 | 정보 탭 탐색             | `tab_click`             |

## 2. 사용 도구 및 연결 구성

- **분석 도구**: Google Analytics 4 (gtag.js)
- **측정 ID**: `G-GZ5F1FVYST`
- **배포**: Vercel (`travlight.vercel.app`)
- **측정 ID 주입**: 코드에 하드코딩하지 않고 **Vercel 환경변수**로 주입
  - Key: `VITE_GA_MEASUREMENT_ID` / Value: `G-GZ5F1FVYST` / Environment: Production
  - ⚠️ Vite는 **빌드 시점**에 환경변수를 넣으므로, 추가/변경 후 **반드시 재배포**해야 반영된다.

### 코드 구조

| 파일                                     | 역할                                         |
| ---------------------------------------- | -------------------------------------------- |
| `src/lib/analytics.ts`                   | GA4 초기화 + 커스텀 이벤트 전송 헬퍼         |
| `src/main.tsx`                           | 앱 부트 시 `initAnalytics()` 호출(태그 로드) |
| `src/components/landing/LandingPage.tsx` | CTA 클릭 시 `trackEnter(source)`             |
| `src/App.tsx`                            | 탭 변경 시 `trackTabClick(tab, country)`     |

측정 ID가 없으면(로컬) 아무것도 전송하지 않는다. → **커스텀 이벤트는 프로덕션에서만 집계.**

## 3. 수집 이벤트 정의

### 자동 수집 (GA4 기본)

`page_view`(조회), `first_visit`(첫 방문), `session_start`(세션 시작)

### 커스텀 이벤트

| 이벤트      | 발생 시점                | 매개변수  | 값                                                                   |
| ----------- | ------------------------ | --------- | -------------------------------------------------------------------- |
| `cta_enter` | 랜딩 → 서비스 진입(전환) | `source`  | `hero` · `final_cta` · `nav`                                         |
| `tab_click` | 진입 후 정보 탭 클릭     | `tab`     | `visa` · `prohibited` · `airline` · `packing` · `transport` · `tips` |
|             |                          | `country` | 선택한 나라 이름                                                     |

- `source` — hero(히어로 CTA) · final_cta(하단 CTA) · nav(상단 바로 시작)
- `tab` — 관심 정보 카테고리 / `country` — 나라별 수요

## 4. GA4 설정 (콘솔)

### 4-1. 맞춤 측정기준 (필수)

관리 → 맞춤 정의 → "맞춤 측정기준" 탭 → 맞춤 측정기준 만들기

| 측정기준 이름 | 범위   | 이벤트 매개변수 |
| ------------- | ------ | --------------- |
| CTA 위치      | 이벤트 | `source`        |
| 탭            | 이벤트 | `tab`           |
| 국가          | 이벤트 | `country`       |

- 이벤트 매개변수 철자는 코드가 보내는 이름과 정확히 일치해야 함. 범위는 모두 **이벤트**.
- ⚠️ **등록 이후 데이터부터** 쪼개져 보임(소급 불가), 처리에 최대 24~48h.

### 4-2. 주요 이벤트(전환) 표시 (선택)

관리 → 이벤트 목록에서 `cta_enter`의 "주요 이벤트로 표시" 토글 ON.

## 5. 분석 방법

- **도달**: 실시간 / 이벤트의 `page_view`·방문자 수
- **전환 + CTA 비교**: 탐색 → 유입경로 탐색 (`page_view → cta_enter → tab_click`), 세분화에 CTA 위치(source)
- **참여**: 탐색 → 자유 형식 (행=탭/국가, 값=이벤트 수, 필터=이벤트 이름 `tab_click`)

필터는 필수 아님. 특정 이벤트만 볼 때만 **이벤트 이름**으로 필터.

## 6. 지표 → 가설 매핑

| 가설 단계     | 지표                    | 보는 곳         |
| ------------- | ----------------------- | --------------- |
| 도달          | 방문자 수, `page_view`  | 실시간 / 이벤트 |
| 전환율        | `cta_enter` ÷ 방문자    | 유입경로 탐색   |
| CTA 효율      | `cta_enter` × `source`  | 유입경로 세분화 |
| 탐색 깊이     | `tab_click` 수          | 자유 형식       |
| 콘텐츠 관심도 | `tab_click` × `tab`     | 자유 형식       |
| 국가 수요     | `tab_click` × `country` | 자유 형식       |

## 7. 셋업 중 이슈 & 배운 점

- **측정 ID/속성 불일치** — 이벤트가 안 잡히던 근본 원인. 코드는 정상이었고, 실시간을 보던 속성과 전송 대상 측정 ID가 어긋나 있었다.
- **Vercel Web Analytics 커스텀 이벤트는 Pro 전용** → 무료로 퍼널 계측하려 GA4로 전환.
- **VITE\_ 환경변수는 빌드타임 주입** → 변경 후 재배포 필수.
- **광고/추적 차단기·백신·보안 DNS** → 테스트 환경에서 GA 요청이 503 등으로 막히면 전송 안 됨. 차단 없는 환경에서 검증.
- **맞춤 측정기준은 소급 불가**, **커스텀 이벤트는 프로덕션에서만**.

## 9. 1차 실험 결과 (2026-07-20)

첫 배포 후 인스타그램·에브리타임 등에 링크를 공유해 수집한 초기 데이터.
**표본이 작고(실사용자 ~30명) 지인 중심의 따뜻한 트래픽**이라, 절대 수치보다 방향성 해석에 무게를 둔다.

### 유입 채널

- 주력: 인스타그램(`l.instagram.com`), 2위: 에브리타임(`everytime.kr`)
- 제외: `Prineville`·`Forest City` + `facebook.com` = 페이스북 크롤러 봇(실사용자 아님)

### 퍼널 요약 (대략)

| 단계 | 지표           | 값                  |
| ---- | -------------- | ------------------- |
| 도달 | 실사용자(추정) | ~30명               |
| 전환 | `cta_enter`    | ~29회               |
| 참여 | `tab_click`    | 다수(1인당 여러 탭) |

### 전환 위치(source)별 `cta_enter`

| CTA 위치         | 횟수(대략) |
| ---------------- | ---------- |
| hero (상단)      | ~21        |
| final_cta (하단) | ~4         |
| nav (바로 시작)  | ~2         |

→ **상단 hero CTA가 전환의 ~70%.** 사용자는 랜딩을 다 안 읽고 위에서 바로 진입.

### 탭별 관심도 (`tab_click`)

| 탭                   | 횟수(대략) |
| -------------------- | ---------- |
| 반입규정(prohibited) | ~27        |
| 항공사(airline)      | ~27        |
| 날씨·패킹(packing)   | ~24        |
| 교통(transport)      | ~19        |
| 꿀팁(tips)           | ~17        |
| 비자(visa)           | ~14        |

⚠️ 비자가 최저지만, **비자가 첫 화면 기본 탭이라 "클릭 없이" 노출되어 과소 집계**된 영향이 큼.

### 국가별 수요 (`tab_click`)

| 국가            | 상대 규모          |
| --------------- | ------------------ |
| 일본            | 압도적 1위 (~60여) |
| 미국            | 2위 (~28)          |
| 베트남          | 소수 (~13)         |
| 프랑스 / 필리핀 | 소수 (~8~9)        |
| 이탈리아 / 호주 | 미미 (~2~4)        |

### 핵심 인사이트

1. **상단 hero CTA가 전환을 이끈다** → 히어로 카피·버튼 강화, 하단은 보조.
2. **수요가 일본·미국에 쏠림** → 커버리지 확장보다 일본·미국 정보 깊이부터 강화.
3. **반입규정·항공사가 최다 클릭** → 여행자 불안 포인트일 가능성. 전면 배치 검토.
4. **비자 탭 과소 집계** → 기본 탭 구조 탓. 초기 탭 노출 별도 측정 또는 기본 탭 변경 후 재비교.

### 해석 시 주의

- 표본이 작고(~30) 따뜻한 트래픽 → 콜드 유입으로 재검증 필요.
- 봇 트래픽 제외, `(not set)` 행은 맞춤 측정기준 등록 이전 데이터.

### 다음 액션

- 표본 확대로 콜드 트래픽에서도 전환·참여 유지되는지 확인.
- 비자 기본 탭 이슈 보정 후 실제 관심도 재측정.
- 일본·미국의 반입규정/항공사 정보 심화.

### 결론

초기 데이터로 **"정보가 흩어져 불편 → 한곳에 모으면 진입·탐색한다"는 가설이 지지**됨.
성공 기준(전환 ≥30%, 참여 ≥60%)을 넘는 것으로 보이며, **표본 확대**로 검증을 단단히 할 단계.
